# 02 · 前端迁移(React / Vite)

> 本章把 `silan-viking/11` 的 schema 变更 + `01-后端迁移.md` 的 API 契约
> 变化,下沉成 **React 前端逐 API 模块 / 逐类型 / 逐组件的改动计划**。
>
> 前端栈:Vite + React 18 + TS + React Router v6 + Context API + i18next。
> API 层在 `frontend/src/api/`,类型在 `frontend/src/types/` 与组件内联。
>
> **每条改动标 `[M0.5a]` / `[M0.5b]`**,对齐 `00-总览.md` §0.3 时序。
> **后端契约真相源是 `01-后端迁移.md`** —— 本章只说前端怎么跟。
>
> **契约维护补充**:React 的 TS 类型不会由 goctl 自动生成。每次改
> `backend/api/backend.api`,都要手动同步 `frontend/src/types/` 与
> `frontend/src/api/`,并用 L4 契约测试证明 JSON 响应字段仍匹配。详见
> `04-工程审查补充.md` §4.4。

## 2.1 [M0.5a] episode 从 blog 的子类型拆成独立 content type

**现状**(探查):episode 不是独立类型 —— 它是 `BlogData.type` 的枚举值
`'episode'`,靠 `BlogData.seriesId` 触发 `SeriesDetailLayout`。series 数据
经 `fetchSeriesData(seriesId)` 单独取。没有 `/episodes` 路由。

schema 把 episode 独立成 type 后,前端要拆:

**API 层** —— 新建 `frontend/src/api/episodes/episodeApi.ts`:

```ts
// 对接 01-后端迁移.md §1.5 的新 endpoint
fetchEpisodeSeriesList()                  → GET /api/v1/episodes/series
fetchEpisodeSeries(seriesSlug)            → GET /api/v1/episodes/series/:slug
fetchEpisode(slug)                        → GET /api/v1/episodes/:slug
recordEpisodeView(id)                     → POST /api/v1/episodes/:id/views
```

`api/blog/blogApi.ts` 里的 series 相关函数(`fetchSeriesData`、
`updateSeriesProgress`、`setCurrentEpisode`)删除;`episodeApi.ts` 只对接新的
episode 读取端点,不保留 blog-series 兼容入口。

**类型层**:

- 新建 `frontend/src/types/episode.ts`(或 `components/` 下):`Episode`、
  `EpisodeSeries` 独立接口。当前 `BlogData` 里的 `seriesId`/`seriesTitle`/
  `episodeNumber`/`totalEpisodes`/`seriesImage` 等 episode 元数据字段 ——
  从 `BlogData` 移到 `Episode`。
- 现 `api/blog/blogApi.ts` 的 `SeriesEpisode`/`SeriesData` 接口 —— 迁到
  episode 类型文件、改名对齐(`EpisodeSeries`)。

**路由 + 页面**(`App.tsx`):

```
新增:  /episodes              → EpisodeSeriesGallery(列容器系列)
       /episodes/:seriesSlug  → EpisodeSeriesDetail(一个系列 + episodes)
       /episodes/:seriesSlug/:episodeSlug → EpisodeDetail(单集)
```

**组件**:

- `components/BlogStack/SeriesDetailLayout.tsx` —— 迁成 episode 专属组件
  (`components/Episodes/EpisodeSeriesDetail.tsx`),不再是 blog 的一种布局。
- `components/BlogStack/BlogDetail.tsx` 里「`if (blog.seriesId)` →
  `SeriesDetailLayout`」的分支 —— **删掉**。blog 详情页只剩
  `ArticleDetailLayout`(article/vlog/tutorial/podcast)。episode 走独立
  路由和组件,不再混进 `BlogDetail`。
- `BlogStack.tsx` 列表 —— 不再混入 episode/series 卡片。

## 2.2 [M0.5a] `blog.content_type` 去 `episode` / `series`

`components/BlogStack/types/blog.ts` 当前:

```ts
type?: 'article' | 'vlog' | 'tutorial' | 'podcast' | 'episode' | 'series';
```

`silan-viking/10`§10.4.2 裁决 `blog.content_type` = `article/podcast/vlog/
tutorial`(4 值)。改为:

```ts
type?: 'article' | 'vlog' | 'tutorial' | 'podcast';   // 去掉 episode / series
```

`BlogData` 里所有 episode/series 元数据字段(§2.1 列的)随之移除 —— 它们
归 `Episode` 类型。所有 `blog.type === 'episode'` / `'series'` 的判断代码
删除(它们现在走 episode 独立路由)。

## 2.3 [M0.5b] resume 从 8-section 聚合改 `part_entry` 读取

**现状**(探查):`api/home/resumeApi.ts` 的 `fetchResumeData()` 一次性取
8 个 section(personal/education/experience/research/publications/awards/
skills/recent),每个 section 是强类型数组(`EducationItem[]` 等)。
`ResumeWebsite.tsx` 消费整个 `ResumeData`。

`silan-viking/11` §11.9 把 education 等表删掉、resume 走 `part_entry`。
`01-后端迁移.md` §1.6:后端 `getresumedatalogic.go` 改读 `part_entry`,
`ResumeData` DTO 形状变(section 不再是强类型数组,是「Part + entry 列表」)。

**这是前端改动最大的一块,排 M0.5b**(M6 验证 part_entry 重建后,与后端
§1.6 同步切)。

**API 层**(`api/home/resumeApi.ts`):

- `fetchResumeData()` —— endpoint 不变(`/api/v1/resume`),但返回结构变。
- 细粒度函数不得再访问旧 endpoint。若内部仍保留函数名供旧组件编译,
  也必须从 `GET /api/v1/resume` 的 `parts` 派生,不得调用
  `/resume/education` 等旧 URL。`fetchPersonalInfo` 保留(`personal_info`
  表留)。
- `fetchRecentUpdates` —— recent_updates 转 `update` type,改调
  `api/updates/`(见 §2.5)。

**类型层**(`types/api.ts`):

- B3 已裁决为「透传 `part_entry` payload」。前端 resume section 从通用
  `ResumePart { role, shape, entries: ResumeEntry[] }` 派生渲染期类型;
  `EducationItem`/`ExperienceItem` 等只作为组件渲染适配层类型,不再对应
  后端 DTO。
- `ResumeData` —— section 字段从 `education: EducationItem[]` 改成
  `parts: ResumePart[]` 或 `Record<string, ResumeEntry[]>`。
- `RecentUpdate` —— 见 §2.5。

> **当前实现选择**:不兼容旧 resume 子端点;`resumeApi.ts` 只读
> `/api/v1/resume` 和 `/api/v1/updates`。

**组件层**:

- `views/ResumeWebsite.tsx` —— 消费 `ResumeData` 的方式改:从「读 8 个强
  类型 section」改成「遍历 `parts`,按 `role` + `shape` 渲染」。
- `components/Resume/OptimizedResumeComponent.tsx` —— 它现在示范细粒度
  加载(分别调 `fetchEducation` 等)。细粒度 endpoint 删了 —— 这个组件
  要么删,要么改成「`fetchResumeData` 一次取、组件内按 Part 懒渲染」。
- `components/Resume/Timeline.tsx`、`RecentSection.tsx` —— 接收的数据
  类型变,跟着 `ResumeData` 改。

## 2.4 [M0.5a] 新增 content 关系 / 交互数据查询

`silan-viking` 的 `content_relation`(演化链)和 `content_interaction`
(浏览/点赞/访客分析)是**新能力**。前端当前**没有**通用的关系展示或
交互数据组件(探查:只有 project↔blog、series↔episode 这种点对点隐式
关系)。

`[M0.5a]` 可选地新增(取决于 silan 要不要现在做前端展示):

- `api/relations/relationApi.ts` —— 查某 Item 的演化链
  (`GET /api/v1/relations?from=...`,对接后端 `content_relation`)。
- 演化链展示组件 —— `VISION.md` §3.2 的「访客顺着 idea→blog→project 链
  浏览」。这是终局画面的一部分,但**不是 M0.5 阻塞项**。
- 交互数据:`content_interaction` 的统计前端,对应 `silan-viking` 需求
  `#15`。同样不阻塞 M0.5。

> **判断**:`content_relation`/`content_interaction` 的前端展示是**新
> 功能**,不是 schema 迁移的「修复项」。M0.5a 把后端表和 API 备好即可;
> 前端展示组件可以排在 M0.5 之后、作为独立 feature 做。本章把它列出来
> 是为了让计划完整 —— 但标注它**不阻塞 M0.5 上线**。

## 2.5 [M0.5a] `update` 成独立 type

`silan-viking/11` §11.7.1 + `01` §1.8:`recent_updates` 转正为 `update`
content type,后端新建 `/api/v1/updates/` handler 组。

前端:

- 新建 `api/updates/updateApi.ts`:`fetchUpdates()` →
  `GET /api/v1/updates/`、`fetchUpdate(slug)` → `GET /api/v1/updates/:slug`。
- `views/RecentUpdates.tsx` —— 现在从 `fetchResumeData().sections.recent`
  取数据。改成调 `fetchUpdates()`。
- `RecentUpdate` 类型 —— 加 `slug`/`visibility`/`update_type` 字段(对应
  schema)。`update_type` 是新的 8 值 enum(`silan-viking/10`§10.4.6)。
- `components/Resume/RecentSection.tsx` —— 若 resume 页仍要嵌「最近动态」,
  它改调 `fetchUpdates()`(而不是从 resume 数据里取);或 resume 页不再
  嵌 update,RecentSection 只在 `/recent-updates` 页用。

## 2.6 [M0.5a/b] project views/likes/graph + comment 契约

**views/likes**(`api/projects/projectApi.ts`):

- `recordProjectView`/`likeProject`/`getProjectMetrics` —— endpoint 不变,
  但后端实现从 `project_views`/`project_likes` 表改成 `content_interaction`
  (`01` §1.3)。**前端 API 函数签名和返回结构若不变,前端无需改** ——
  后端保持 `getProjectMetrics` 返回 `{likes_count, views_count,
  is_liked_by_user}` 的契约即可。`02` 这里只需**契约验证**(`03` 联调),
  不一定有代码改动。
- `ProjectDetail.tsx` 消费 `metrics` 的代码 —— 若契约不变,不改。

**graph**(`fetchGraphData` → `/api/v1/projects/graph`):

- 后端 `getprojectgraphdatalogic.go` 从 `project_relationships` 改读
  `content_relation`(`01` §1.6)。若后端保持 `GraphData` 返回结构不变,
  前端 `fetchGraphData` 和图组件不改。`[M0.5b]`(随后端 §1.6)。

**comment**(三处 comment 类型):

- `BlogCommentData`/`ProjectCommentData`/`IdeaCommentData` —— 后端
  `comment.referrence_id` → `reference_id`(`01` §1.2)。若前端 comment
  类型里**没有**直接映射 `referrence_id` 字段(探查显示前端 comment 类型
  用的是 `id`/`content`/`likes_count` 等,未见 `referrence_id`)——
  前端 comment 类型**可能无需改**。`03` 联调时核对。

> **本节多数是「契约验证」而非「代码改动」** —— 只要后端保持 API 的
> 请求/响应 JSON 形状不变(尽管底层表换了),前端就不用动。`03-联调与
> 切换.md` 的契约对齐验证专门钉这件事:**哪些是后端内部重构(前端透明)、
> 哪些是真的契约变更(前端必改)**。
