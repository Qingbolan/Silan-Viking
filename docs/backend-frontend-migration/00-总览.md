# 00 · 前后端迁移总览

> **这个目录是什么**:`silan-viking` 引擎的 M0.5 改了数据库 schema
> (`docs/silan-viking/11-M0.5-ent-schema-PR.md`)—— 删 17 张表、加 9 张表、
> 改 5 张表。这个 schema 变更**连锁到 Go 后端和 React 前端**:后端 18 个
> handler / 14 个 logic 会编译失败或语义错位,前端 6 个 API 模块 / 一批
> 类型与组件的契约要跟着改。
>
> `11` 只用「ent 重新生成 + Go API 适配」一句带过这件事。**这个目录补上
> 那份缺失的、可编码级的前后端更新计划。**
>
> | 文件 | 内容 |
> |---|---|
> | `00-总览.md`(本文)| 迁移背景、与 `silan-viking/11` 的关系、时序、一图看全 |
> | `01-后端迁移.md` | 逐 handler / 逐 logic 的 Go 后端改动计划 |
> | `02-前端迁移.md` | 逐 API 模块 / 逐类型 / 逐组件的 React 前端改动计划 |
> | `03-联调与切换.md` | 前后端联调顺序、契约验证、灰度切换、回滚 |
> | `04-工程审查补充.md` | 未落地缺口、goctl/ent 生成流程、数据采集/维护边界 |

## 0.1 为什么需要这个目录 —— schema 变更不止是「重新生成 ent」

`silan-viking/11` 的 M0.5 是这样描述后端工作的:

> ent 重新生成 → Go API 适配 → promote 改造 → `sea-orm-cli` 反向生成。

「Go API 适配」五个字背后,是探查 `backend/` 后的真实账单:

- **18 个 handler** 直接读写了会被删/改的表。
- **14 个 logic 文件** 含对这些表的 ent 查询 —— ent 表删掉后,这些
  `client.Education.Query()` 之类的调用**直接编译失败**。
- **4 个 `types.go` 的 DTO struct**(`Award`/`Education`/`WorkExperience`/
  `Publication`)镜像了被删的表 —— 要删。
- **API 路由**有 4 个 resume 子 endpoint(`/resume/education` 等)对应被删表。

前端那边:

- **episode 当前不是独立类型** —— 它是 `BlogData.type` 的一个枚举值
  (`'episode'`),靠 `seriesId` 字段触发 `SeriesDetailLayout`。schema 把
  episode 独立成 type 后,前端的类型、路由、组件都要拆。
- **resume 当前一次性读 8 个 section**(`fetchResumeData` 聚合
  education/experience/publications/awards/...)—— schema 把这些表并成
  `part_entry` 后,简历页的读取契约整个变。
- `blog.content_type` 前端类型里含 `'episode'`/`'series'` —— 要去掉。

这些不修,M0.5 之后**后端编译不过、前端白屏**。这个目录是这件事的施工图。

## 0.3 迁移如何对齐 `silan-viking` 的 M0.5a / M0.5b 时序

`silan-viking/11` §11.12 把 M0.5 拆成两段(silan [P1] 裁决):**M0.5a 加新表
保留旧表 → M6 验证 `part_entry` 重建 → M0.5b 删旧表**。前后端迁移**必须挂在
这个时序上**,不能自己另立一套。

```
M0.5a ──────────────────────────────────────────────┐
  后端:加新表 ent + 改 enum + 新表的读写 handler      │
        旧表 handler 暂时保留(此刻旧表还在)          │
  前端:新增 episode/关系/交互 的 API 与组件           │
        简历页暂时仍走旧 8-section endpoint            │
                                                       │
        ↓ M1–M6:Rust 引擎实施,M6 跑 sync 验证          │
        ↓ part_entry 能从新结构正确重建                 │
                                                       │
M0.5b ──────────────────────────────────────────────┤
  后端:删旧 resume 表 ent + 删对应 4 handler/4 logic/ │
        4 DTO;resume endpoint 改读 part_entry          │
  前端:简历页切到 part_entry 读取契约,删旧 8-section  │
        调用                                            │
                                                       │
  ⚠️ M0.5b 必须在 M6 之后 —— 否则删了旧 resume 表,    │
     新的 part_entry 还没被 Rust sync 验证过,简历      │
     数据断档(silan [P1] 裁决)。                       │
└──────────────────────────────────────────────────────┘
```

**这个目录的三份计划据此分阶段标注**:`01`/`02` 每条改动都标 `[M0.5a]` 或
`[M0.5b]`,`03` 给联调与切换的具体顺序。

> **一条不可动摇的边界**:`project_views`/`project_likes`/`comment_likes`
> 和 `comment` 是**运行时数据**(线上访客产生)。删它们 / 改它们的迁移
> SQL **在生产服务器执行一次、不可重跑**,且 promote job 永不碰运行时表
> (`silan-viking/11` §11.11、`08` §8.3)。`03` 专门讲这条的施工纪律。

## 0.4 这个目录与 `silan-viking/` 的关系

- `silan-viking/11` 是 **schema 真相源**(ent 表怎么改)。本目录**不重复**
  字段定义 —— 要看表结构去 `11`。
- 本目录是 **schema 变更的下游施工图**:`11` 说「ent 这么改」,本目录说
  「于是 Go handler 这么改、React 组件这么改」。
- `silan-viking/05`(测试)的 **L4 契约测试**是前后端契约对齐的验收闸门 ——
  `03-联调与切换.md` 对接它。
- 时序锚点是 `silan-viking/04` 里程碑的 M0.5a/M0.5b/M6。

> 读法:先读 `silan-viking/11` 知道 schema 怎么变,再读本目录 `01`(后端)
> 和 `02`(前端)知道代码怎么跟,再读 `04` 补齐生成链路与数据维护边界,
> 最后 `03` 知道怎么联调上线。
