## 3.1 联调前必须先拍的裁决

`01`/`02`/`04` 写作时挖出 6 个需 silan 拍的点。**它们不拍,联调没法开始** ——
因为它们决定 API 契约的形状。

| # | 来源 | 缺口 | 不拍的后果 |
|---|---|---|---|
| B1 | `01` §1.9 | comment 点赞落 `content_interaction` 还是留 `comments.likes_count` | 三个 `like*commentlogic.go` 无法重写 |
| B2 | `01` §1.9 | `blog_series` 表去留 | episode 迁移范围不定 |
| B3 | `01` §1.9 / `02` §2.3 | resume API 透传 `part_entry` payload vs Go 组装强类型 | 前端 resume 类型无法定形 |
| B4 | `silan-viking/10` 账本 #9 | project `priority`/`complexity` 引不引入 | `projects` 表字段不定(小项)|

| B6 | `04` §4.3 | `request_logs`/`user_identities` 是否收敛为 ent schema 单真相源 | 启动 DDL 与 ent schema 双真相源会漂移 |

> 这 6 项**不是凉冰能裁的**(它们是基线之上的取舍 —— `VISION.md` §4

## 3.3 运行时数据迁移 —— 不可逆操作的施工纪律

`project_views` / `project_likes` / `comment_likes` 是访客在**线上**产生的
真实历史;`comment` 表也是。这些表的迁移**在生产服务器执行、不可重跑**。
施工纪律(承 `silan-viking/11` §11.9、`08` §8.3):

**迁移脚本三条硬规则**:

1. **幂等保护**。`project_views → content_interaction` 是 INSERT-SELECT。
   重跑会重复插入、污染计数。脚本开头检查一次性标记(如
   `migration_log` 表里有无 `m05b_runtime_migration` 记录),已执行则退出。
2. **先备份**。执行前 `cp portfolio.db portfolio.db.pre-m05b`。运行时
   数据无法从 markdown 重建(它不是派生物)—— 备份是唯一回路。
3. **drop 在 INSERT 验证之后**。先 INSERT-SELECT 进 `content_interaction`、
   `SELECT count(*)` 两边核对行数一致,**再** drop 旧表。不能 INSERT 和
   drop 写在同一个不检查的事务里。

**迁移脚本骨架**(M0.5b,生产服务器):

```sql
BEGIN;
-- 0. 幂等检查(伪代码:脚本外层判断,已执行则不进入)

-- 1. project_views → content_interaction
INSERT INTO content_interaction
  (id, entity_type, entity_id, kind, user_identity_id,
   fingerprint, ip_address, user_agent, session_duration, created_at)
SELECT id, 'project', project_id, 'view', user_identity_id,
       fingerprint, ip_address, user_agent, session_duration, created_at
FROM project_views;

-- 2. project_likes → content_interaction(kind='like')
INSERT INTO content_interaction (...) SELECT ..., 'like', ... FROM project_likes;

-- 3. comment_likes → 取决于 B1 裁决:
--    若 B1=(a) 加 entity_type='comment':INSERT 进 content_interaction
--    若 B1=(b) 留 comments.likes_count:不迁明细,comment_likes 直接 drop
--      (likes_count 计数列在 comment 主表已有)

-- 4. 行数核对(脚本层 assert 两边一致)

-- 5. 标记一次性完成
INSERT INTO migration_log (name, ran_at) VALUES ('m05b_runtime_migration', now());
COMMIT;

-- 6. 核对通过后,单独事务 drop
DROP TABLE project_views;
DROP TABLE project_likes;
DROP TABLE comment_likes;       -- B1=(b) 时
DROP TABLE project_relationships;
```

> `comment.referrence_id → reference_id` 列改名(M0.5a 阶段⑥):
> `ALTER TABLE comments RENAME COLUMN referrence_id TO reference_id;`
> —— 是列改名,不是 drop+add,**不丢数据**。但仍是运行时表上的一次性
> DDL,先备份。

**派生表 vs 运行时表 —— promote 永不混**:简历专用表(~17 张)、
`blog_posts`/`ideas`/`projects` 的 enum 改 —— 这些是**派生表**,
`silan index sync` 能重建,promote job 整体删+重建它们(`silan-viking/11`
§11.11 白名单)。运行时表(`content_interaction`/`comment`/`annotation`/
`user_identities`/`request_logs`)promote **一个都不碰**。这条边界是
deploy 安全的根 —— 详见 `silan-viking/08` §8.3。

## 3.4 契约对齐验证 —— 区分「后端内部重构」与「真契约变更」

`02` §2.6 反复出现一句话:很多前端「改动」其实是后端内部换表、对前端
**透明**。联调时必须把这两类分清,否则会改不该改的前端代码。

**判据**:看 API 的**请求/响应 JSON 形状**变没变。

| API | 底层表变化 | JSON 契约 | 前端要改吗 |
|---|---|---|---|
| `GET /projects/:id/metrics` | `project_likes` → `content_interaction` | `{likes_count, views_count, is_liked_by_user}` 不变 | **否**(后端透明)|
| `POST /projects/:id/view` | 同上 | 请求/响应不变 | **否** |
| `GET /projects/graph` | `project_relationships` → `content_relation` | `GraphData` 结构若保持 | **否** |
| comment CRUD | `referrence_id` → `reference_id` | 若前端类型没映射该字段 | **否**(核对确认)|
| `GET /resume` | 8 张表 → `part_entry` | `ResumeData` 结构**变了** | **是**(M0.5b)|
| episode | 从 blog 拆出 | 新 endpoint、新结构 | **是**(M0.5a)|
| `GET /resume/education` 等 | 表删 | endpoint **没了** | **是**(删调用)|

**契约对齐的验收闸门 = `silan-viking/05` 的 L4 契约测试**。`05` 的 L4
本来是「Rust sync 产出的 db 与 Go ent / 前端读取契约逐项对齐」。本次迁移
后,L4 契约测试要**覆盖到**:每个 API endpoint 的响应 JSON ↔ 前端 TS
类型逐字段对齐。联调阶段⑨和⑭跑 L4,全绿才算契约对齐。

> **联调具体做法**:后端 M0.5a 部署到联调环境后,前端**先不改**、直接
> 跑现有前端 —— 凡是「后端透明」的 API,前端应当照常工作;凡是真契约
> 变更的,前端会报错。报错点 = 真正要改前端的点。这一步把「内部重构」
> 和「真契约变更」用运行结果分开,比逐文件猜准。

## 3.5 灰度与回滚

**这是单租户、单实例项目**(`silan-viking` 需求 `#13`)—— 没有多实例
灰度的余地。切换策略相应简单,但回滚必须可靠。

**切换**:
- M0.5a 后端是 additive(加表为主)+ 少量改 —— 部署风险低,后端先上、
  观察、再上前端 M0.5a。
- M0.5b 是删表 —— 风险集中在这。M0.5b 后端 + 前端**一起切**(resume
  契约变了,两端不能错峰)。

**回滚**:
- M0.5a 回滚:additive 改动天然好回滚;`comment` 列改名要有反向 SQL
  (`reference_id` → `referrence_id`),备份在手。
- M0.5b 回滚:**这是真正的风险点**。简历专用表 drop 后,回滚 = 从
  `portfolio.db.pre-m05b` 备份恢复。运行时数据迁移后,`content_interaction`
  里有了新行 —— 回滚要么整库恢复备份,要么删掉迁进去的行(靠 created_at
  / 迁移批次标记定位)。**所以 §3.3 规则 2「先备份」是 M0.5b 回滚的唯一
  保险**。

**回滚判据**:M0.5b 切换后,跑 `silan-viking/05` 的 L4 契约测试 + 前端
冒烟(简历页、blog 页、episode 页、project 页能正常渲染)。任一不过 ——
回滚到备份,排查,不带病上线。
