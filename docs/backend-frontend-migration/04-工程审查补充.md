## 4.1 未落地问题清单

| 优先级 | 缺口 | 为什么危险 | 补充设计 |
|---|---|---|---|
| P0 | `.api` / goctl 只说了“改后重新生成”,没说生成命令、生成物边界、删 endpoint 后的 orphan 文件清理 | goctl 不会替你推理旧 endpoint 是否该删;旧 handler/logic 留在仓里会让迁移看起来完成但代码实际漂移 | 见 §4.2 |
| P0 | ent 只说“重新生成”,没说删 schema 后如何清 stale ent 文件 | 删除 `education` 等 schema 后,旧 generated package 可能仍残留;代码还能误引用旧 ent 类型 | 见 §4.3 |
| P0 | `content_interaction` 的 like/view 语义没定死 | 当前 `project_likes` 是“当前点赞状态”,`project_views` 是“事件日志”;合一后若都按“一行一事件”算,unlike 会把计数算错 | 见 §4.5 |
| P0 | 数据采集/维护链路没有端到端责任边界 | Rust sync、promote、Go API、前端谁写哪张表没钉死,容易在 deploy 时覆盖运行时数据 | 见 §4.6 |
| P1 | `request_logs`/`user_identities` 仍有手写 DDL 与 ent schema 双真相源风险 | `servicecontext.go` 现在启动时手建表;M0.5 后 request_logs 要正式 ent 化,不能继续双写 schema | 见 §4.3 |
| P1 | 前端 TS 类型仍是手写,没有和 `backend.api` 的契约验证机制 | goctl 只生成 Go DTO,不会生成 React 类型;后端契约变了前端可能静默漂移 | 见 §4.4 |

## 4.2 `.api` / goctl 的落地流程

当前仓库的 API 真相源是:

```
backend/api/backend.api
```

goctl 生成物是:

```
backend/internal/types/types.go
backend/internal/handler/routes.go
backend/internal/handler/<group>/*handler.go
backend/internal/logic/<group>/*logic.go
```

本仓当前生成头是 `goctl 1.8.4`;迁移期间先固定用这个版本,不要混用不同
goctl 版本制造大面积无意义 diff。

```bash
cd backend
goctl --version
goctl api go -api api/backend.api -dir .
go test ./...
```

**新增 endpoint 的步骤**:

1. 在 `backend.api` 的 `type (...)` 块里补 request / response DTO。
2. 在对应 `@server` group 里补 route;新资源例如 episode/update 应建新
   group,不要塞进 blog/resume。
3. 运行 `goctl api go -api api/backend.api -dir .`。
4. 实现新生成的 `internal/logic/<group>/*logic.go`。
5. `go test ./...`。若只是生成了 handler 但 logic 没实现,不能算完成。

**删除 endpoint 的步骤**:

1. 从 `backend.api` 删除 route 和不再使用的 DTO。
2. 运行 goctl,让 `routes.go` 和 `types.go` 收敛。
3. 手动删除 orphan 文件:旧 `handler/<group>/*handler.go`、旧
   `logic/<group>/*logic.go`。goctl 生成不会可靠清理这些旧文件。
4. 用 `rg` 证明旧 endpoint/DTO 不再被引用:

```bash
rg -n "GetEducation|Education|WorkExperience|Publication|Award" internal api
go test ./...
```

**不要手改生成物**:`types.go`、`routes.go`、handler 文件都必须通过
`backend.api` 生成。业务分支、查询、事务、错误语义只放在 logic。

## 4.3 entgo 的落地流程

当前 ent 真相源在:

```
backend/internal/ent/schema/
```

生成命令:

```bash
cd backend
go run -mod=mod entgo.io/ent/cmd/ent generate ./internal/ent/schema
go test ./...
```

**M0.5a 加表**:

1. 新增 schema 文件,如 `contentrelation.go`、`contentinteraction.go`、
   `itempart.go`、`partentry.go`、`episode.go`。
2. 运行 ent generate。
3. additive-only 迁移可用:

```bash
go run ./cmd/migrate -db-driver sqlite3 -db-source "/path/to/portfolio.db?_fk=1"
```

**M0.5b 删表/删 schema**:

1. 先改 logic,让业务代码不再引用旧 ent client,例如
   `DB.Education.Query()`、`DB.ProjectLike.Query()`。
2. 删除 schema 文件。
3. 删除对应 generated ent 残留文件和 package。不要假设 ent generate 会替
   你清干净旧实体。
4. 运行 ent generate。
5. 用 `rg` 证明旧实体彻底消失:

```bash
rg -n "Education|WorkExperience|Publication|Award|ProjectView|ProjectLike|CommentLike|ProjectRelationship" internal/ent internal/logic
go test ./...
```

**双真相源必须消掉**:`internal/svc/servicecontext.go` 当前启动时手写
`request_logs` 和 `user_identities` DDL。M0.5 把 `request_logs` 正式 ent
化后,这段手写 DDL 必须删除或只保留兼容迁移脚本,不能让 ent schema 和启动
DDL 同时定义同一张表。`user_identities` 已有 ent schema,也应收敛到同一
迁移路径,避免字段/索引漂移。

## 4.4 前后端契约维护

goctl 只生成 Go 的 `internal/types/types.go`;React 的 TS 类型仍是手写:

```
frontend/src/types/api.ts
frontend/src/api/**/*
```

所以每次改 `backend.api` 必须同步维护一张契约对照表:

| 契约项 | 后端源 | 前端落点 |
|---|---|---|
| resume 返回结构 | `backend.api` 的 `ResumeData` | `frontend/src/types/api.ts` + `api/home/resumeApi.ts` |
| episode DTO | `backend.api` 新 episode type | `frontend/src/types/episode.ts` + `api/episodes/episodeApi.ts` |
| update DTO | `backend.api` 新 update type | `frontend/src/types/api.ts` 或 `types/update.ts` + `api/updates/updateApi.ts` |
| project metrics | `ProjectMetricsResponse` 等 | `api/projects/projectApi.ts` |

契约变更的验收不是“前端 build 过了”而已,还要至少保存每个关键 endpoint
的 JSON 样例并用 L4 契约测试比对字段。手写 TS 类型可以存在,但必须有
测试证明它仍贴着 `backend.api`。

## 4.5 `content_interaction` 的工程语义

`content_interaction` 不应简单理解为“所有交互一行一事件”。当前系统里:

- `project_views` 是事件日志,且有“一小时内同一用户/指纹去重”的业务规则。
- `project_likes` / `comment_likes` 是当前状态表,同一用户或指纹对同一对象
  只能有一条,unlike 会删除这条记录。
- `blog_posts.like_count` 当前只是计数列,没有身份去重,迁移后必须补齐
  toggle 语义,否则比 project 弱。

M0.5 建议采用以下统一规则:

| kind | 行语义 | 写入规则 | 计数规则 |
|---|---|---|---|
| `view` | 事件日志 | 插入事件;同一 `entity_type/entity_id/actor` 一小时内可由 logic 去重 | `count(kind=view)` |
| `like` | 当前状态 | like 时插入;unlike 时删除;同一 `entity_type/entity_id/actor/kind` 唯一 | `count(kind=like)` |

actor 的归一化顺序:

1. 有 `user_identity_id` 用它。
2. 否则用 `fingerprint`。
3. 两者都没有时,公开 API 不接受 like;view 可以降级为 IP+UA 统计,但不参与
   like toggle。

这要求 `content_interaction` 至少有以下索引设计:

```text
index(entity_type, entity_id, kind)
index(entity_type, entity_id, section_anchor)
index(created_at)
```

like 状态还需要唯一性,但**不能**直接加全表
`unique(entity_type, entity_id, kind, actor)` 后把 view 事件也限制住。实现
方式二选一:

- 数据库支持 partial unique 时,只对 `kind='like'` 建唯一约束。
- 跨 SQLite/MySQL/Postgres 保持一致时,在 Go logic 里用事务做“先查再插/删”,
  并把并发冲突当成幂等成功处理。

> 这会新增一个裁决点 B5:`content_interaction.kind=like` 是状态行还是事件行。
> 本章建议“like=状态行、view=事件行”,与当前项目点赞语义一致。

## 4.6 数据采集与维护边界

迁移后必须按下面这张表维护数据,不能混写。

| 数据 | 真相源 | 写入者 | 读取者 | deploy/promote 规则 |
|---|---|---|---|---|
| blog/project/idea/episode/update/resume 内容 | `content/SCHEMA.md` + `content/resources/**` | Rust `silan index sync` | Go API、React | promote 可删重建派生表 |
| `item_part` / `part_entry` / translations | Rust parser 从 Part 文件生成 | Rust sync | Go API resume/detail logic | 派生表,可重建 |
| `content_relation` | frontmatter/SCHEMA 关系声明 | Rust sync | Go API relation/project graph | 派生表,可重建 |
| comment | 线上访客输入 | Go API | Go API、React | 运行时表,promote 永不碰 |
| content_interaction | 浏览/点赞/章节打点 | Go API | Go API stats/metrics、React | 运行时表,promote 永不碰 |
| annotation(reader) | 线上访客/agent 运行时批注 | Go API 或受控 agent endpoint | Go API | 运行时表,promote 永不碰 |
| request_logs | HTTP middleware | Go API | owner-only stats | 运行时表,promote 永不碰 |
| user_identities | OAuth / auth endpoint | Go API | comment/like 归因 | 运行时表,promote 永不碰 |

**新增一种内容字段时**:

1. 若字段是可索引、可筛选的 frontmatter 字段:改 `content/SCHEMA.md`、Rust
   parser/mapper、ent schema、`backend.api` DTO、前端 TS 类型。
2. 若字段只属于 resume entry payload:优先留在 `part_entry.shared_payload` /
   `part_entry_translation.localized_payload`,不改 Go ent,只改 SCHEMA 和前端
   渲染器。
3. 若字段是运行时统计:只能由 Go API 写入运行时表,Rust sync 不得生成。

**新增一个前端展示模块时**:

1. 先确认它读的是派生数据还是运行时数据。
2. 派生数据优先走资源专属 endpoint,例如 `/api/v1/episodes/...`。
3. 运行时统计走 metrics/stats endpoint,不要让前端直接知道底层表。
4. 前端只维护 `frontend/src/api/<domain>/` 和 `frontend/src/types/` 的契约,
   不把数据库表结构泄漏到组件。
