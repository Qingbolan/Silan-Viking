## 1.4 [M0.5a] `blog_posts` / `ideas` / `projects` 的 enum 与 visibility

`silan-viking/11` §11.7:三张主表删 `is_public`、加 `visibility` enum;
`blog_posts.content_type` 改 4 值(去 episode 加 podcast/tutorial)、删
`ideas_id` 外键。

| 受影响 logic | 改动 |
|---|---|
| `logic/blog/getblogpostslogic.go` / `getblogpostlogic.go` / `getblogpostbyidlogic.go` / `searchblogpostslogic.go` | 所有 `blog_posts.IsPublicEQ(true)` 过滤 → `blog_posts.VisibilityEQ(blogpost.VisibilityPublic)`;`content_type` 过滤值去掉 `episode` |
| `logic/ideas/getideaslogic.go` / `getideahandler.go` / `searchideaslogic.go` / `getideacategorieslogic.go` | `idea.IsPublicEQ(true)` → `idea.VisibilityEQ(idea.VisibilityPublic)` |
| `logic/projects/getprojectslogic.go` / `getprojectdetaillogic.go` / `searchprojectdetailslogic.go` | `project.IsPublicEQ(true)` → `project.VisibilityEQ(project.VisibilityPublic)` |

> **语义注意**:旧 `is_public` 是 bool(true/false 两态),新 `visibility`
> 三态(`private`/`unlisted`/`public`)。「对公众可见」的判断从
> `is_public==true` 变成 `visibility=='public'`(`unlisted` 不进公开列表
> 但有直链可访问 —— Go API 的列表 endpoint 过滤 `public`,详情 endpoint
> 允许 `public`+`unlisted`)。这是**语义变更**,不是字段改名,每个读这两
> 张表的 endpoint 都要按这条重新想「这里该露 public 还是 public+unlisted」。

`ideas_id` 外键删除:`blog_posts` 不再直接关联 idea。原本靠
`blog_posts.ideas_id` 表达的「这篇 blog 由某 idea 演化而来」,迁到
`content_relation`(`from_type=blog, to_type=idea, relation_type=evolved_into`)。
受影响:任何 JOIN `blog_posts.ideas_id` 的 logic(探查未发现现役 logic
直接用它,但 `getprojectrelatedblogslogic.go` 类逻辑要核)。

## 1.5 [M0.5a] episode 独立成 type —— 新 handler/logic 组

`silan-viking/11` §11.5.2:episode 是独立 `episodes` + `episode_series`
表,不再是 `blog_posts` 的行。当前后端 episode 是怎么处理的?探查发现:
episode 走 **blog 的 series 机制** —— `blog/getblogserieslogic.go` +
`fetchSeriesData` endpoint,episode 是 `blog_posts` 里 series 下的行。

M0.5a 要新增 episode 的独立 handler 组(改 `backend.api` 新增 endpoint):

```
GET  /api/v1/episodes/series                  → 列容器系列
GET  /api/v1/episodes/series/:series_slug      → 一个系列 + 它的 episodes
GET  /api/v1/episodes/:slug                    → 一个 episode
POST /api/v1/episodes/:id/views                → episode 浏览打点(content_interaction)
```

对应新建 `logic/episodes/` 组(自底向上,go-zero `goctl` 生成 handler
骨架,logic 手写读 `episodes`/`episode_series` ent)。

**旧 blog series 机制的处置**:`blog/getblogserieslogic.go` 当前服务
「blog 的系列」。`silan-viking/11` §11.5.2 的注里留了个待确认点 ——
`blog_series` 表若 blog 还有「松散博文系列」用法则保留,否则删。

> ⚠️ **列入 silan 裁决清单(§1.9)**:`blog_series` 表 + `getblogserieslogic.go`
> 去留 —— 取决于 blog 是否要保留独立于 episode 容器系列的「博文系列」。
> 探查显示前端 `fetchSeriesData`/`SeriesDetailLayout` 目前是给 episode 用
> 的;若 episode 全迁到独立 type,`blog_series` 很可能可删。在裁定前,
> `getblogserieslogic.go` 不动。

## 1.7 [M0.5a→M0.5b] 数据库迁移脚本 —— additive 迁移器做不了的部分

| `comment.type`/`entity_type` 改 enum | M0.5a | 生产服务器 | 否 |
| `blog_posts`/`ideas`/`projects` 删 `is_public`、加 `visibility` | M0.5a | 派生表,可重建,跟 sync 走 | 是 |
| `blog_posts.content_type` enum 改 4 值 | M0.5a | 派生表,但要先确认无 `content_type=episode` 残行(它们已迁 `episodes`)| — |

| drop 简历专用表(~17 张)| M0.5b | 派生表,但删前确认 `part_entry` 重建成功(M6 后)| 否 |
