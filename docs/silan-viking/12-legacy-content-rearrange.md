# 12 · Legacy-content offline rearrange

> This chapter fills a **doc gap** silan pointed out: the design
> pinned "incompatible with the legacy layout", but **no rule** said
> how the old `README.md` / `NOTES.md` / `resume.md` should be
> moved one-shot into the new structure
> `content/resources/{type}/{item}/parts/<role>/`. This chapter is
> that ruleset.
>
> **Status of this chapter**: a one-shot action between M0 and M0.5.
> It is the specification for an **offline rearrange script** — the
> script moves legacy content into the latest layout and is then
> **thrown away**.
>
> **Three iron rules** (per the `04` milestones, `08` §8.1 / §8.7):
> 1. The rearrange is **one-shot**. The runtime parser **has no**
>    legacy-layout compatibility branch.
> 2. The rearrange script reads the legacy structure and writes the
>    new structure; it is **not** part of the silan-viking engine.
>    It can be an independent Python script / tool (it already has
>    to read the old Python repo's layout).
> 3. The rearrange output must pass every `fatal` check in the M0
>    `SCHEMA.md` (`10`) — otherwise the rearrange is not done.

---

## 12.1 The legacy content is not in this working tree — a fact that must be stated upfront

After verifying this repo (`Silan-Personal-Website`): **the legacy
markdown content repo is not in this working tree**. This repo
currently holds the Go backend + the React frontend + the Python
`silan` package; the real ideas/blog/projects markdown is maintained
elsewhere by the content owner.

Therefore the rules in this chapter are **based on the legacy layout
validated by the Python parser** (`silan/parsers/`'s docstrings and
parsing logic are the factual evidence of the legacy content shape —
whatever the parser can parse is what the legacy content really looks
like).

> **One step before executing the M0 rearrange**: silan connects the
> legacy content repo (or points at its path), runs the cross-check
> in §12.6 once, and confirms the real legacy filenames / layout
> match §12.2; on divergence the mapping table in §12.2 is patched on
> the spot. **Without verifying against the real legacy content, the
> rearrange is not done.**

## 12.2 Old layout → new layout mapping table

Old layout (facts extracted from the Python parser docstrings):

```
idea:     <idea-folder>/  README.md  NOTES.md  results.md  REFERENCES.md
                          experiments.md  assets/  .silan-cache
project:  <project-folder>/  README.md  RELEASES.md|CHANGELOG.md
                             QUICKSTART.md  requirements.txt|package.json
blog:     <blog-folder>/  <name>.md  (+ series via folder prefix / frontmatter)
resume:   resume.md  (single file, 7 ## sections: Education / Work Experience /
                      Publications / Awards / Skills / Research / Recent Updates)
```

New layout (`01` §1.3.1 + `10` §10.4):

```
content/resources/{type}/{item-slug}/
├── parts/<role>/meta.toml      # part_id generated by the rearrange script
└── parts/<role>/<lang>.<ext>   # prose uses .md; structured Parts use .toml
```

**Per-type mapping** — which part of which legacy file lands where in the new structure:

### idea

| Old source | New destination | Notes |
|---|---|---|
| `README.md` frontmatter | `parts/overview/<lang>.md` frontmatter | Item-level fields |
| `README.md` body | `parts/overview/<lang>.md` body | abstract → overview Part |
| `NOTES.md` body | `parts/progress/<lang>.md` body | progress Part |
| `results.md` body | `parts/result/<lang>.md` body | result Part |
| `REFERENCES.md` body | `parts/reference/<lang>.md` body | reference Part |
| `experiments.md` | **drop, or merge into progress** | The new SCHEMA has no experiments Part (`10` §10.4.1) — at rearrange time, append into progress or have the human trim |
| Legacy `.silan-cache` | **drop** | The engine regenerates it (`08` §8.2: `.silan-cache` is engine-written) |
| `assets/` | `parts/overview/assets/` or Item-level `assets/` | Image assets travel along |

### project

| Old source | New destination |
|---|---|
| `README.md` frontmatter | `parts/overview/<lang>.md` frontmatter |
| `README.md` body | `parts/overview/<lang>.md` body |
| `RELEASES.md` / `CHANGELOG.md` | `parts/release_notes/<lang>.md` body |
| `QUICKSTART.md` | `parts/quick_start/<lang>.md` body |
| `requirements.txt` / `package.json` | **does not enter content** | A dependency manifest is a code artefact, not content; `license` / `version` go into the overview frontmatter |
| goals / challenges paragraphs inside the README | split into `parts/goals/`, `parts/challenges/`, etc. | If the legacy README writes all of these in one piece, **split manually** at rearrange time (see §12.4) |

### blog

| Old source | New destination |
|---|---|
| `<name>.md` frontmatter | `parts/body/<lang>.md` frontmatter |
| `<name>.md` body | `parts/body/<lang>.md` body |

### episode (independent type — final ruling #1)

> episode is its own type, its own container series (`10` §10.4.4).
> The rearrange target structure is
> `content/resources/episode/<series-slug>/<episode-slug>/parts/body/`,
> not rows in `blog_posts`.

| Old source | New destination |
|---|---|
| Legacy container-series directory (`series/` or `episode-*/`) | `content/resources/episode/<series-slug>/` |
| Legacy series description / config | `<series-slug>/series.toml` (container-series metadata) |
| Each legacy episode file's `episode-NN-*.md` frontmatter | `<episode-slug>/parts/body/<lang>.md` frontmatter, with `series` + `episode_number` |
| Each legacy episode file body | `<episode-slug>/parts/body/<lang>.md` body |

> ⚠️ The old design docs (`06` / `05`) at one point wrote episode as
> "container-series legacy layout `README.md + episode-01-*.md`, no
> parts". **That shape is obsolete** — the rearrange target only
> recognises the new structure above. The old episode descriptions
> in `06` / `05` are corrected in the `15` write-back pass.

### resume (config-driven Parts — final ruling #2)

> Legacy resume is a **single file with 7 sections**; the new resume
> is a **single Item + config-driven Parts** (`10` §10.4.5). Each
> section rearranges into the corresponding Part; entries in
> `entry_list`-shape Parts (education etc.) are parsed into
> schema-validated entries and land in `part_entry`.

| Legacy `resume.md` section | New Part | shape |
|---|---|---|
| File frontmatter + personal info at the top | `parts/summary/<lang>.md` | prose |
| `## Education` | `parts/education/` | entry_list |
| `## Work Experience` | `parts/experience/` | entry_list |
| `## Publications` | `parts/publications/` | entry_list |
| `## Awards` | `parts/awards/` | entry_list |
| `## Research Experience` | `parts/research/` | entry_list |
| `## Skills` | `parts/skills/` | key_value_list |
| `## Recent Updates` | **rearranged into independent `update` Items** | — see below |

> Rearrange of `entry_list` Parts: each `### Institution` sub-section
> under the legacy `## Education` → one entry → parsed by the
> `entry_fields` from `10` §10.4.5. **The output is
> `parts/<role>/<lang>.toml`** (TOML array-of-tables, not `.md`),
> one `[[entry]]` block per entry, fields per `entry_fields`. The
> rearrange script **assigns each entry an `entry_id = "e_<ulid>"`**
> written into the TOML block — the same education entry's `en.toml`
> and `zh.toml` share one `entry_id` (multilingual alignment is by
> id, not by ordinal). Entries that cannot resolve a `required`
> field → marked for manual review (§12.4).
>
> `## Recent Updates` is **no longer dropped** (final ruling #3:
> `update` is the 6th type) — each entry in the legacy resume's
> Recent Updates section is rearranged into an independent `update`
> Item under `content/resources/update/`; see the "update" section
> below.

### update (the 6th type — final ruling #3)

> update is the 6th content type (`10` §10.4.6). Legacy update
> content has two sources: legacy update files handled by the
> Python `update_parser`, and the `## Recent Updates` section of
> legacy `resume.md`.

| Old source | New destination |
|---|---|
| Legacy update files (`YYYY-MM-DD-slug.md` etc.) frontmatter | `content/resources/update/<slug>/parts/body/<lang>.md` frontmatter |
| Legacy update file body | `parts/body/<lang>.md` body |
| Each `### Type: Title` sub-section under the legacy `resume.md` `## Recent Updates` | One independent `update` Item: sub-heading → `title`, `*Type*` → `update_type`, `*Date*` → `date`, body → `parts/body` |

> Legacy update `type` values are normalised to the 8-value
> `update_type` enum from `10` §10.4.6 via the Python
> `update_parser` mapping; values that don't normalise are marked
> for manual review (§12.4).

## 12.3 Language-variant rearrange

How the legacy content's multilingual variants (the
`file_info.language` of the Python parser) land in the new layout:

- Each legacy language version (if the legacy repo distinguishes via
  `README.zh.md` / `README.en.md` or subdirectory) → lands as
  `<lang>.<ext>` (e.g. `zh.md` / `en.md`, or `zh.toml` / `en.toml`)
  under the same `parts/<role>/`.
- `meta.toml`'s `canonical_lang` = the language the legacy content
  marks as primary (default `en`).
- Phase-two translation metadata (`translation_of` / `source_hash` /
  `stale`) is **left empty** (`01` §1.3.2 phase one) — the rearrange
  script does not fill them, only positions the fields.

> If the legacy repo has **no** multilingual distinction (single
> language only), after rearrange the Part has only one
> `<canonical_lang>.md` — this is legal (`10` §10.6
> `info: canonical_lang_only`).

## 12.4 Automatic rearrange vs manual split — draw the line

Not every rearrange step can be scripted. **What scripts can do**
and **what must be manual** must be separated:

**Scripts do automatically**:
- Move files (`README.md` → `parts/overview/en.md`).
- Generate `meta.toml` + `part_id` (`p_<ulid>`).
- Normalise frontmatter field names (old `featured_image` /
  `image` / `banner` → new `featured_image_url`, per the field
  tables in `10`).
- Normalise enum values (old project `status: ACTIVE` → new
  `active`; old idea `status: idea` → new `hypothesis`, per the
  Python parser status mappings).
- Slug validation / generation (when the legacy has no slug, derive
  one from the title per `10`'s slug pattern).

**Must be manual** (the script only marks; it does not guess):
- A legacy project `README.md` that writes overview / goals /
  challenges / solutions **in one continuous markdown file** — the
  script cannot reliably split it into 4 Parts. The script drops
  the whole thing into `parts/overview/` and **marks** "this
  project's README needs a manual Part split" in the rearrange
  report. silan splits it manually afterwards.
- Legacy idea `experiments.md` (the new SCHEMA has no such Part) —
  the script appends it into progress and marks it; silan decides
  whether to keep or delete.
- Any enum mapping that hits the "Default" branch (the legacy value
  cannot be normalised) — marked for manual confirmation.

> **Principle**: the script **prefers marking over guessing**. A
> mis-split Part is harder to spot and harder to fix than a "drop
> everything into overview + mark for manual split".

## 12.5 Rearrange-script artefacts and validation

The script runs in 5 stages, each one loggable, to avoid "discover
the wrong move only after completion":

1. **scan**: read-only against the legacy content repo; produce
   `old-inventory.json`. Record each legacy Item's type, slug,
   candidate languages, legacy file list, and unknown file list.
2. **plan**: per the §12.2 mapping, produce `migration-plan.json`.
   Every legacy file must have one of four actions: `move` /
   `merge` / `drop` / `manual`; unknown files must not be silently
   skipped.
3. **emit**: write the new layout into a temporary directory;
   generate `part_id` / `entry_id`; copy assets. This stage must
   not modify the legacy content repo.
4. **validate**: run the M0 `SCHEMA.md` fatal checks against the
   temporary directory; if it fails, the formal `content/` is not
   overwritten.
5. **promote**: after silan confirms the report, move the temporary
   directory to the formal target path.

After the rearrange script runs, two artefacts are produced:

1. **The new structure under `content/resources/`** — entirely in
   the `parts/<role>/` layout.
2. **The rearrange report `migration-report.md`** listing:
   - "Old path → new path" per Item.
   - All "manual split / confirm" marks (§12.4).
   - Every enum mapping that fell to the Default branch.
   - The dropped-file list (`experiments.md` / legacy `.silan-cache`).
     Note: `## Recent Updates` sections are **not dropped** — they
     rearrange into `update`-type Items.

**Completion criteria for the rearrange** (per `08` §8.7):
- [ ] The rearrange output is placed under
      `engine/tests/fixtures/content/` or the formal `content/`,
      **entirely in the new layout**, with no legacy-path residue.
- [ ] Running the M0 `SCHEMA.md` (`10`) checks against the
      rearrange output: **zero `fatal` errors**. `warn`-level
      issues are allowed but must be listed in the report.
- [ ] Every "manual split" marker in `migration-report.md` is
      **reviewed and processed** (either split, or confirmed not
      to split).
- [ ] The rearrange script itself does **not** enter `engine/` — it
      is a one-shot tool, lives under `tools/` or `scripts/`,
      archived after the run. The runtime parser has no
      legacy-layout branch (`08` §8.1).

## 12.6 Cross-check list for the operator

After silan connects the real legacy content repo, before executing
the M0 rearrange:

- [ ] Do the file names inside a real legacy idea folder match the
      §12.2 idea table? (`README.md` / `NOTES.md` / `results.md` /
      `REFERENCES.md`) — if they diverge, patch the §12.2 mapping.
- [ ] Is the real legacy project README "one continuous piece" or
      "already split into sections"? Determines the scale of
      manual splits in §12.4.
- [ ] How does the legacy repo distinguish multilingual content
      (filename suffix / subdirectory / single language)? Maps to
      §12.3.
- [ ] Does the real legacy `resume.md` have 7 `##` sections, or
      more / fewer? Check against the §12.2 resume table.
- [ ] How are legacy episodes / series organised (container
      directory / `episode-*.md` naming / series config file)?
      Maps to the container-series rearrange in §12.2's episode
      section.
- [ ] Where does legacy update content sit (standalone update
      files / `resume.md`'s Recent Updates section / both)? Maps
      to §12.2's update section.
- [ ] Are there file kinds in the legacy content that §12.2 does
      not cover? If so, patch a mapping line into this chapter on
      the spot.

> The script is touched only after the cross-check list is cleared
> and §12.2's mapping is aligned with the real legacy content.
> Writing the script before cross-checking against real legacy
> content is guessing — and "don't extrapolate from the current
> state" forbids that.
