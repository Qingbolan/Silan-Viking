## 11.2 ➕ `content_relation` —— 通用关系表(修订 A)

承 `01` §1.10 修订 A,**逐字段落 ent**:

```go
// backend/internal/ent/schema/contentrelation.go
type ContentRelation struct{ ent.Schema }

func (ContentRelation) Fields() []ent.Field {
    return []ent.Field{
        field.UUID("id", uuid.UUID{}).Default(uuid.New),
        field.Enum("from_type").
            Values("blog", "project", "idea", "episode", "resume", "update"),
        field.UUID("from_id", uuid.UUID{}),
        field.Enum("to_type").
            Values("blog", "project", "idea", "episode", "resume", "update"),
        field.UUID("to_id", uuid.UUID{}),
        field.Enum("relation_type").
            Values("evolved_into", "documents", "references",
                   "supersedes", "part_of"),
        field.Int("sort_order").Default(0),
        field.Time("created_at").Default(time.Now),
    }
}

func (ContentRelation) Indexes() []ent.Index {
    return []ent.Index{
        index.Fields("from_type", "from_id", "to_type", "to_id",
                     "relation_type").Unique(),   // 去重
        index.Fields("from_type", "from_id"),     // 正向查
        index.Fields("to_type", "to_id"),         // 反向查
    }
}
```

**对齐 `10` 的裁决**:
- `relation_type` enum **只存 canonical 方向** —— `evolved_from` 不入库
  (它是 `evolved_into` 的 flip,`10`§10.5),所以 enum 里没有 `evolved_from`。
- `from_type`/`to_type` enum **含全部 6 个 type:`blog/project/idea/
  episode/resume/update`** —— 最终裁决 `update` 是第 6 种 content type
  (`10`§10.4.6 裁决 #3),`update` 是合法的 relation endpoint(一条 update
  可以 `documents` 一个 project、`evolved_into` 边的一端等)。
  (注:早期草案曾写「去掉 update」,那基于 update 不是 type 的旧判断,
  已被裁决 #3 推翻 —— 此处与 Go schema 的 6 值 enum 一致。)

### 11.5.1 ➕ `part_entry` + `part_entry_translation`(裁决 #2)

承 `10` §10.4.5 最终裁决 #2:resume 的结构化 Part(`entry_list` shape)
**不为每种 Part 建专用 ent 表**,统一落通用 `part_entry`。

```go
// partentry.go —— 一行一条 entry_list 条目(一条 education / 一条 experience)
func (PartEntry) Fields() []ent.Field {
    return []ent.Field{
        field.UUID("id", uuid.UUID{}).Default(uuid.New),
        field.UUID("item_part_id", uuid.UUID{}),  // 隶属哪个 entry_list Part
        field.String("entry_id"),                 // e_<ulid>,源 TOML 来,稳定
        field.Int("sort_order").Default(0),
        // 语言无关字段(date/url/logo/bool…),按 SCHEMA entry_fields
        //   中 translatable=false 的字段;是 SCHEMA-validated typed JSON,
        //   不是无约束 blob —— sync 写库前必按 entry_fields validate。
        field.JSON("shared_payload", map[string]any{}),
        field.Time("created_at").Default(time.Now),
    }
}
func (PartEntry) Indexes() []ent.Index {
    return []ent.Index{
        index.Fields("item_part_id"),
        index.Fields("item_part_id", "entry_id").Unique(),  // entry 稳定锚点
    }
}
```

> **`entry_id` 是 entry 的稳定锚点**(评审 P1 裁决,见 `10`§10.4.5):
> 它之于 entry 等同 `part_id` 之于 Part —— 引擎生成、终生不变、写进源
> TOML。有它,sync 能精确定位「哪一条 entry 变了」,不必 delete+insert
> 整个 Part;agent 改单条 education 的 `git diff` 也干净。`(item_part_id,
> entry_id)` 唯一 —— 一个 Part 内 entry_id 不重复。

```go
// partentrytranslation.go —— 条目的语言相关字段
func (PartEntryTranslation) Fields() []ent.Field {
    return []ent.Field{
        field.UUID("id", uuid.UUID{}).Default(uuid.New),
        field.UUID("part_entry_id", uuid.UUID{}),
        field.String("language_code"),
        // 语言相关字段(title/details/description…),entry_fields 中
        //   translatable=true 的字段;同样是 SCHEMA-validated typed JSON。
        field.JSON("localized_payload", map[string]any{}),
        field.Time("created_at").Default(time.Now),
    }
}
func (PartEntryTranslation) Indexes() []ent.Index {
    return []ent.Index{
        index.Fields("part_entry_id", "language_code").Unique(),
    }
}
```

> **payload 不是无约束 blob**(最终裁决 #2 的硬约束):`shared_payload` /
> `localized_payload` 的 JSON 结构由 `SCHEMA.md` 对应 Part 的 `entry_fields`
> 定义。Rust sync 写库前**必须**按 `entry_fields` validate —— 违反则
> `10` §10.6 的 `entry_field_violation`(fatal)。
>
> **扩展性收益**:加一种 resume Part(`talks`/`patents`/`service`)= 在
> `SCHEMA.md` 加一段 `entry_fields` 配置。`part_entry` 表**不动**、无新表、
> 无迁移、无 mapper 改动。这是裁决 #2 的根本目的。
>
> **未来若需高频查询**某个 entry 字段(如按 `start_date` 排序所有 entry):
> 二期加 projection / materialized view,**不**为此把 `part_entry` 拆成宽表
> (最终裁决 #2 原文:「不提前为所有 resume 块上宽表」)。

## 11.11 派生表 / 运行时表白名单(对接 `08` §8.3 promote)

`08` §8.3 要求 deploy promote **只替换派生表,不碰运行时表**。M0.5 必须
把每张表明确归类,产出这份白名单 —— promote job 按它执行。

> **白名单有两个版本** —— 因为 M0.5 拆成 M0.5a / M0.5b(§11.12),简历
> 旧表在 M0.5a 期间还在、M0.5b 才删。promote 改造在 M0.5a 落地,所以
> M0.5a 期间 promote 必须用**过渡态白名单**,M0.5b 后切到**终态白名单**。

**派生表 —— M0.5a 过渡态白名单**(M0.5a ~ M0.5b 之间,promote 按此):

```
内容主表:   blog_posts, ideas, projects, personal_info, recent_updates,
            episodes, episode_series
translation: *_translations(全部,含 episode/episode_series 的)
Part 正文:   item_part, item_part_translation
Part 条目:   part_entry, part_entry_translation   ← resume 的 entry_list
关系:        content_relation
注册:        sync_meta
★ 简历旧表:  education, education_details, work_experience,
            work_experience_details, publications, publication_authors,
            awards, social_links（+ 各 translation）
            —— M0.5a 期间这些表仍存在且仍是派生表,promote 照旧重建它们。
               原因:M0.5a 没删它们(§11.12),sync 此时还可能在写它们。
```

**派生表 —— M0.5b 终态白名单**(M0.5b drop 简历旧表后,promote 切到此):

```
内容主表:   blog_posts, ideas, projects, personal_info, recent_updates,
            episodes, episode_series
translation: *_translations(全部)
Part 正文:   item_part, item_part_translation
Part 条目:   part_entry, part_entry_translation
关系:        content_relation
注册:        sync_meta
（简历旧表 education/work_experience/... 已 drop,不在终态白名单)
```

> **切换时机**:M0.5b drop 简历旧表的同一个 PR 里,promote job 的派生表
> 白名单从过渡态改成终态 —— 删表和改白名单必须同一次提交,否则 promote
> 会 `DELETE FROM education` 一张不存在的表而报错。
> `recent_updates` 两个版本都在(它是 `update` type 的内容主表,裁决 #3)。

**运行时表**(Go API 在访客访问时写;promote **绝不触碰**):

```
content_interaction        ← 浏览/点赞打点
comment                    ← 评论
annotation(author_kind=reader 的行)  ← 访客批注
user_identities            ← 访客身份
request_logs               ← API/访问日志(裁决 #7,独立正式表)
```

> **边界微妙处**:`annotation` 一张表里**两类数据混存** —— `author_kind=owner`
> 的批注是 owner 直接写的(可进 Git/可派生),`author_kind=reader` 是运行时。
> **M0.5 裁决**:`annotation` 整表归**运行时表**(promote 不碰)。owner 批注
> 若要进版本管理,二期单独设计;M0.5 不为这个边角拆表。
>
### promote job 实现级合同(M8 核对,M9 实现)

promote job 的 SQL 只允许按下列顺序生成,不能用整库文件替换:

```
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
BEGIN IMMEDIATE;
DELETE FROM <derived_table_1>;
DELETE FROM <derived_table_2>;
...
INSERT INTO <derived_table_1> ...;
INSERT INTO <derived_table_2> ...;
...
INSERT INTO sync_meta(key, value) VALUES('content_commit', ?)
  ON CONFLICT(key) DO UPDATE SET value = excluded.value;
COMMIT;
```

- `<derived_table_*>` 只能来自本节过渡态/终态白名单。运行时表名一个都不许
  出现在 promote SQL 里。
- `BEGIN IMMEDIATE` 先拿写锁;Go API 读侧走 WAL,读不被长时间阻塞。若
  `busy_timeout` 到期,本次 promote 失败退出,线上 DB 保持旧状态。
- 所有 `DELETE` 与 `INSERT` 在同一事务内;任一表写失败则 rollback,不得留下
  半新半旧的派生表。
- `sync_meta.content_commit` 最后更新,它是“这批派生表完整可用”的提交标记。
  监控/回滚只看这个 commit,不看中间表行数。
