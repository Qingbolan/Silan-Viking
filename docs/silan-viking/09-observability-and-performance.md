# 09 · Observability, error handling, performance

> Engineering-lead addition (晓雯). The previous eight chapters fully
> cover **functional correctness** (objects, four-layer tests, the
> cross-check), but a system that will run a long-lived MCP server,
> do git operations, and be called frequently by agents is **flying
> blind without observability** and **will hit the wall during
> implementation without a performance budget**. This chapter is the
> runtime contract that must be nailed before M1 Rust work starts.
>
> Three things: ① structured logging / tracing; ② layered error
> handling; ③ scale assumptions + performance budget.

## 9.1 Error handling — typed-error layering (decide before M1)

`01` §1.5.1 already uses the `ParseError` / `Issue` baseline, but
there is no **global error contract**. Pinning it:

### 9.1.1 One error enum per crate, derived via `thiserror`

| Crate | Error type | Typical variants |
|---|---|---|
| `silan-viking-base` | `BaseError` | `InvalidUri`, `HashMismatch` |
| `silan-viking-content` | `ContentError` | `MalformedManifest`, `OrphanPart`, `DanglingRelation` |
| `silan-viking-app` | `ParseError` / `SyncError` / `ProposalError` | `MissingRequiredPart{role}`, `SchemaViolation{file,line}`, `MergeConflict{files}`, `StaleProposal{base,head}` |
| `silan-viking-entities` | (generated by sea-orm; not hand-written) | — |
| Three L4 adapters | `CliError` / `McpError` / `SiteError` | wrap the lower-layer error + adapter context |

**Rules**:
- Library crates (L1–L3) always return `Result<T, ThisCrateError>`; **never use `anyhow`**.
- `anyhow` is **only used at the L4 adapter process boundary** (`main.rs` / top-of-MCP-handler) — to turn typed errors into the final human / agent message.
- **Zero `unwrap()` / `expect()` in non-test code** (`clippy` set to `-D unwrap_used`). The sole exception: unrecoverable config errors at process start, `expect` with a clear message.
- Every error variant must carry **locating + actionable** info: `SchemaViolation` carries `file:line`, `MergeConflict` carries the list of conflicting files — no `Other(String)` allowed.

### 9.1.2 Exit-code mapping (carries from `06` §6.8)

`06` §6.8 fixes three exit-code classes `0/1/2`. The error-enum-to-exit-code mapping happens in the L4 adapter:

```
ContentError / ParseError (user content / config issue, fixable)  → exit 1
SyncError::DbUnwritable / environment-class (git missing, network down) → exit 2
ProposalError::MergeConflict / StaleProposal                       → exit 1 (with a next-step hint)
panic (a bug that shouldn't happen)                                → exit 101 (Rust default)
```

`panic` is not used as a normal error path; once a `panic` happens, there is a bug, and a panic in CI = test failure.

## 9.2 Structured logging and tracing — runtime observability

### 9.2.1 Technology choices

- `tracing` + `tracing-subscriber`: structured span / event; not `println!`.
- Two output forms: human-readable for dev (`tracing-subscriber` fmt); JSON lines for `--json` / the MCP server (collected by a log system).
- Level convention: `ERROR` for operation failures; `WARN` for non-fatal anomalies (dangling edges, stale translations); `INFO` for the start / end of key operations; `DEBUG` for per-Item details; `TRACE` for per-Part/File details.

### 9.2.2 Span contract for core operations

**Each core operation is a `#[tracing::instrument]` span with fixed
fields**. That way any failure / slow path can be located from the
span tree down to "which operation, which Item, which Part".

| Operation | Span name | Required fields | Sub-spans |
|---|---|---|---|
| `silan index sync` | `sync` | `content_root`, `mode` (full/incremental) | one `sync.item{uri, kind}` per Item; under it `parse.part{role, lang}` |
| `silan proposal accept` | `proposal.accept` | `proposal_id`, `base_oid` | `merge`, `validate`, `update_ref{old, new}` |
| `silan proposal propose` (MCP) | `proposal.propose` | `proposal_id`, `touched_parts` | `validate` |
| `silan site deploy` | `site.deploy` | `target_host`, `dry_run` | `sync`, `build`, `package`, `ship`, `promote` |
| Each MCP request | `mcp.request` | `tool`, `agent_id`, `namespace` | the operation span for that tool |
| `silan stats *` | `stats.query` | `endpoint`, `target_host` | — |

**At span end, always record**: `elapsed_ms`, outcome (`ok` / `err` + the error variant name). The root span of `sync` additionally records: `items_scanned`, `items_changed`, `items_skipped`.

### 9.2.3 Observable artefacts of sync — `content/log.md` + sync_meta

- `content/log.md` (`00` §0.4 already has it): append-only, one line per sync / accept — `## [ISO8601] sync | +3 ~1 -0 | commit <sha>` — a human-readable operation history.
- The `sync_meta` table (`08` §8.3): `content_commit` + `synced_at` + `duration_ms` + `items_total`. `silan index status` reads it to answer "when was the last sync, how big".
- Together, silan doesn't need to grep logs to know how the system has been running.

### 9.2.4 MCP server observability (long-lived process, the key focus)

The MCP server is the only long-lived process; blind spots there are the most dangerous:

- Each agent connect / disconnect: one `INFO` line (`agent_id`, the SCHEMA version pushed in the handshake).
- Each tool call: an `mcp.request` span — which agent, what was called, elapsed, outcome.
- `silan mcp status` shows not just "is it running" but: current connection count, per-agent call totals, the most recent error.
- Every `ctx_write` / `reflect` write to `agent/` gets `INFO` with fields: `agent_uri`, `commit_oid`, `content_hash`, `mode`, `written_files`. Consistent with the agent-write git-commit discipline in `03`: direct writes are not trace-less.

## 9.3 Scale assumptions — without pinned scale, every performance judgment downstream is a castle in the air

silan-viking is a **one-person** context system (`#13`). Define scale
upper bounds accordingly (M1 implementation designs for this scale —
not over-engineered for millions):

| Dimension | Expected scale | Design upper bound (implementation stress-tests to this) |
|---|---|---|
| Content items (blog + idea + project + episode + resume) | tens to hundreds | 1,000 items |
| Parts per Item | 1–4 | 8 |
| Languages | 2 (en/zh) | 4 |
| Total `content/` repo size | tens of MB | 500 MB (including assets) |
| `portfolio.db` | a few MB to tens of MB | 200 MB |
| Concurrent agent proposals | 1–2 | 5 pending proposals |
| `content_interaction` rows (on server) | grows with visits | considered separately, see below |

> `content_interaction` (the ping table) is the only table that
> **grows unbounded** — it sits on the server and accumulates with
> visits. It does not affect local sync performance (the local side
> doesn't hold it, `08` §8.3), but the server side needs a
> **periodic archive / aggregation policy** — to be decided when
> M0.5 edits ent: raw rows kept N months, older rows rolled into a
> daily-aggregate table. Listed separately here; not part of the
> local engine's scale budget.

## 9.4 Performance budget — acceptable elapsed for each operation

Based on §9.3's design upper bound (1000 items), pin the
**performance budget**. This is the acceptance gate, not a wish:

| Operation | Budget (1000-item upper bound) | What if it overruns |
|---|---|---|
| `silan index sync` full | < 5 s | over → parallelise the parser (no inter-Item dependency, naturally parallelisable) |
| `silan index sync` incremental (one Item changed) | < 200 ms | over → check whether hash comparison is really incremental |
| `silan index rebuild` | < 8 s | same as full sync + derived-layer rebuild |
| `silan <type> show` / `list` | < 100 ms | querying portfolio.db; indexes should exist |
| MCP `recall` (semantic / lexical) | < 500 ms | lexical fallback is the default offline path; Embedder unavailability must not drag `recall` down |
| MCP `read` / `ctx_read` | < 50 ms | single-file read |
| `proposal accept` (merge + validate) | < 3 s | validation ② is the bulk; can validate only the merge-affected Items |
| `silan stats *` (remote) | < 1 s | network-bound; budget includes one HTTP round-trip |

**Two design decisions forced by the budget**:
1. Full sync budget 5s / 1000 Items — Items have no inter-parsing dependency, so **the parser must be parallelisable** (`rayon` or tokio tasks). This must be left at §1.5 `Parser` design time: `parse` is a pure function with no shared mutable state, naturally parallelisable. M1 implements for parallelism.
2. Incremental sync budget 200ms — `.meta.toml` / `.silan-cache` hash comparison must be O(changed) not O(total). `sync` first reads the `.silan-cache` hash manifest and re-parses only the Items whose hash changed.
3. `recall` budget 500ms — M7 defaults to a lexical index; optional Embedder is only a recall enrichment. On model / network failure, degrade and record the span field `embedder=fallback`; do not return 5xx. Also record `embedder=none|api|local|fallback` and `recall_mode=lexical|hybrid`; with no Embedder, do not advertise "semantic recall enabled" in logs, CLI, or skill text. The experience ceiling of lexical fallback is written into `01` §1.5.Q: it only guarantees keyword hits; it does not guarantee recognising the same idea rephrased.

## 9.5 Performance benchmarks — criterion + CI regression gate

- The L1–L4 tests (`05`) verify **correctness**; benchmarks verify **non-regression of performance**.
- Tool: `criterion`. Benchmark points: full `sync` (100-Item / 1000-Item fixture, two tiers), incremental `sync`, `recall`, `accept`.
- Bench fixtures: outside `engine/tests/fixtures/`, build a separate `engine/benches/fixtures/` that synthesises 100 / 1000-Item content repos (script-generated, deterministic).
- **CI regression gate**: benchmark results stored as a baseline; if a new commit pushes any metric `> baseline × 1.5` → CI warns; `> ×2` → CI fails. The baseline is updated for deliberate performance improvements.
- The bench job is independent of `05`'s L1–L4 and does not block functional CI, but must be green before merging to main.
